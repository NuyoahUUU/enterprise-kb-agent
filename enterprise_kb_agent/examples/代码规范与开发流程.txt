公司内部代码规范与开发流程 v3.0
文档编号：PROC-2026-001
生效日期：2026-01-01

一、代码规范

Python 规范：
1. 使用 Python 3.10+ 版本，新项目推荐 3.11
2. 代码格式化使用 Black（line-length=100），导入排序使用 isort
3. 类型注解强制：所有函数参数和返回值必须标注类型
4. 文档字符串使用 Google Style Docstring
5. 命名规范：类名大驼峰，函数/变量小写下划线，常量全大写下划线
6. 单个函数不超过 60 行，单个文件不超过 500 行
7. 禁止使用裸 except，必须指定异常类型

Java 规范：
1. 使用 JDK 17 LTS，Spring Boot 3.x
2. 代码格式化使用 Google Java Format
3. 禁止使用 Lombok 的 @Data 注解（会导致循环引用问题），改用 @Getter/@Setter
4. Controller 层只做参数校验和路由，业务逻辑全部在 Service 层
5. 禁止在 Mapper 中写复杂 SQL，复杂查询使用 MyBatis-Plus 的 LambdaQueryWrapper

数据库规范：
1. 所有表必须有 id（BIGINT 自增主键）、create_time、update_time 三个字段
2. 索引命名：唯一索引 uk_表名_字段名，普通索引 idx_表名_字段名
3. 禁止使用外键约束，关联关系在应用层维护
4. VARCHAR 字段必须指定长度，禁止使用 TEXT 代替大字段（除非确实超 65535 字符）
5. SQL 查询必须使用参数化，严禁字符串拼接

Git 规范：
1. 分支策略：main → develop → feature/xxx, bugfix/xxx
2. Commit Message 格式：type(scope): description
   类型：feat, fix, docs, refactor, test, perf, chore
3. 合并到 main 必须通过 Code Review（至少 1 人 approve）
4. 禁止 force push 到共享分支

二、开发流程
1. 需求评审：产品输出 PRD，技术评审通过后进入开发
2. 技术方案：涉及数据库变更、新服务、架构调整的必须先写技术方案并评审
3. 开发阶段：在 feature 分支开发，单元测试覆盖率 ≥ 70%
4. Code Review：提交 Pull Request，至少一人 approve
5. 测试阶段：QA 在 staging 环境验证，通过的 feature 合并到 develop
6. 上线流程：每周二、四 16:00 发布窗口，发布前需回滚方案
7. 上线后监控：灰度上线 10% → 观察 30 分钟 → 全量
